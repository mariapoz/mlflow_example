# **Отчет по домашней работе №2**

## 1. Сформулировать существующие проблемы, зафиксировать их.
   
   Исходное состояние: централизованное файловое хранилище (каталог models/), куда разные ML-команды выгружают результаты обучения в виде папок с артефактами. Стандарта на структуру и имена директорий нет; версии, датасеты и прочие признаки кодируются произвольно в названии. Метаданные (автор, дата, метрики, параметры, коммит, окружение) не сохраняются и не индексируются.
   
**Проблемы**

1. Отсутствие метаданных (неизвестны автор, датасет, метрики, параметры, код/окружение)
2. Нет версионирования как сущности (версии зашиты в названия папок, нет связи и истории)
3. Нет управления жизненным циклом (непонятно, что в prod/staging)
4. Нет поиска и discoverability (нельзя быстро найти нужную модель/версию по команде, задаче, тегам)
5. Нет контроля доступа и аудита (нет ролей, нет истории кто/что менял)
6. Технические риски хранения (нет целостности/проверок, возможны битые/неполные артефакты)
7. Нет интерфейса интеграции (нет стандартного API/CLI для деплоя и пайплайнов)

## 2. На их основе продумать функциональные и нефункциональные требования к системе хранения моделей, зафиксировать эти требования.

**Функциональные требования**

- Хранить артефакты модели - модель должна лежать в одном месте (папка/архив в storage), чтобы её можно было скачать/развернуть
- Загружать (регистрировать) модель - команда должна уметь добавить новую модель/версию в реестр и привязать к ней артефакты
- Отдавать модель по запросу - по запросу система должна возвращать нужную модель (ссылку на артефакты + метаданные)
- Поддерживать версионирование - для одной модели должны существовать версии (v1, v2, …), чтобы было понятно, что новее/старее и что именно используется
- Понимать, с каким датасетом связана версия - для версии фиксируем датасет (id/версия/ссылка), чтобы можно было воспроизвести обучение и объяснить результат
- Хранить параметры модели - для версии сохраняем основные параметры обучения/конфига (hyperparameters, настройки пайплайна и т.п.)
- Хранить метрики модели - для версии сохраняем метрики качества (accuracy, AUC, RMSE и т.д.), чтобы сравнивать версии между собой
- Иметь “ярлыки” окружений: prod / stage / ab - должен быть способ пометить, какая версия сейчас в проде, какая в стейдже, какая участвует в A/B
- Понимать "кто это сделал" - логируем автора и изменения: кто загрузил версию, кто перевёл в prod, кто поменял ярлыки
  
**Нефункциональные требования**

- Надёжно: метаданные не теряются, операции атомарные  
- Безопасно: доступ по ролям/командам, чужие модели не редактируются  
- Удобно для интеграции: простое API для обучения/деплоя  
- Масштабируемо: список/поиск моделей работает нормально при росте количества версий  
- Расширяемо: можно добавлять новые поля метаданных без ломания схемы (например через JSON)  
- Прозрачно: аудит/логи изменений, чтобы разбирать инциденты и понимать историю

## 3. На основе требований спроектировать архитектуру системы, отобразить ее в виде схемы, обосновать все компоненты и выбор технологий.
Общая идея: делаем централизованный Model Registry: артефакты моделей лежат в storage (FS), а вся информация о моделях хранится в базе данных и доступна через API.

**Компоненты**

1) Registry API (backend-сервис)

- единая точка входа для регистрации моделей, назначения `prod/stage/ab`, поиска и выдачи “правильной” версии;
- централизованная бизнес-логика (валидации, права доступа, аудит).

Обоснование:  
- легко интегрируется с пайплайнами обучения/CI, деплоем и сервисами инференса;
- стандартный формат (JSON), понятные контракты, легко тестировать.

2) База данных метаданных (PostgreSQL)

- хранить сущности: модели, версии, алиасы (prod/stage), теги, пользователей, аудит;
- обеспечивать уникальность и целостность (например, alias не может указывать на несуществующую версию).

Обоснование: 
- надёжный "де-факто стандарт" для таких систем;
- поддерживает JSONB (удобно хранить расширяемые метаданные: метрики/параметры без постоянных миграций);
- легко запускается в Docker Compose.

3) Хранилище артефактов (FS)

- хранить реальные файлы моделей: веса, конфиги, токенайзеры, препроцессинг и т.д.

Обоснование: 
- модели уже лежат на конкретной машине в папках;
- минимальная сложность: registry хранит `artifact_uri` как путь к папке.

4) Аудит (Audit Log)

- понимать, кто зарегистрировал версию, кто перевёл в prod, кто поменял alias;
- разбирать инциденты и обеспечивать “историю решений”.

Обоснование: 
- события не должны теряться;
- удобно строить отчёты и расследования.

5) Авторизация и роли (RBAC + JWT)

- несколько команд работают на одной системе, нужно разграничивать доступ;
- предотвращение случайных/неправильных действий (перевод в prod не тем человеком).

Обоснование: 
- простой способ аутентификации для сервисов и пайплайнов;
- не требует серверных сессий, удобен для API.

Почему такой набор компонентов
- API + DB решают главную проблему хаоса: появляется единый источник правды о версиях и метаданных.
- Shared FS позволяет быстро закрыть задачу "с нуля" без переезда артефактов и сложной инфраструктуры.
- Архитектура расширяема: FS можно заменить на MinIO/S3 без изменения основной модели данных.

## 4. Продумать API и схему БД.
**API**
- `POST /auth/login` — получить токен доступа
- `POST /models` — создать карточку модели
- `GET /models` — список/поиск моделей (по имени, тегам, команде)
- `GET /models/{name}` — получить модель и список версий
- `POST /models/{name}/versions` — зарегистрировать новую версию
- `GET /models/{name}/versions/{version}` — получить конкретную версию
- `POST /models/{name}/aliases/{alias}` — назначить alias на версию (`prod/stage/ab`)
- `GET /models/{name}/resolve?alias=prod` — получить “актуальную” версию по alias
- `GET /health` — healthcheck

**БД (таблицы)**
**teams**
- `id`, `name`

**users**
- `id`, `email`, `password_hash`, `role`, `team_id`, `created_at`

**models**
- `id`, `name` (unique), `team_id`, `description`, `tags`, `created_at`

**model_versions**
- `id`, `model_id`, `version` (int), `artifact_uri`, `checksum`, `size_bytes`,
  `params` (json), `metrics` (json), `dataset_ref`, `commit_sha`,
  `created_by`, `created_at`

**aliases**
- `id`, `model_id`, `alias` (`prod/stage/ab`), `version_id`,
  `updated_by`, `updated_at`
- unique constraint: `(model_id, alias)`

**audit_log**
- `id`, `actor_id`, `action`, `entity_type`, `entity_id`, `payload` (json), `created_at`
